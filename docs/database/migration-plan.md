# Database Migration Plan

**Project:** AI-SDLC Working Office  
**Version:** 1.0.0  
**Date:** 2026-06-18  
**Tool:** Alembic (Python)  
**Target Database:** PostgreSQL 15+

---

## Overview

All database schema changes are managed through **Alembic** versioned migrations. No manual SQL changes to production are allowed. Every schema change — including new tables, added columns, enum type changes, and index additions — must go through a migration file.

---

## Setup

```bash
# Install Alembic
pip install alembic

# Initialise Alembic in the backend folder
cd backend
alembic init alembic

# alembic.ini: set sqlalchemy.url
# sqlalchemy.url = postgresql+psycopg2://user:password@localhost:5432/ai_sdlc
```

**`alembic/env.py`** must import all SQLModel table definitions so auto-generation works:

```python
from models import *  # noqa — import all SQLModel tables
target_metadata = SQLModel.metadata
```

## Naming Convention

| Object | Convention | Example |
|---|---|---|
| Migration file | `{revision_id}_{short_description}.py` | `0001_initial_schema.py` |
| Table | `snake_case`, plural | `pipeline_steps` |
| Column | `snake_case` | `project_id` |
| Index | `idx_{table}_{column(s)}` | `idx_documents_project_id` |
| FK constraint | `fk_{table}_{column}_{ref_table}` | `fk_tasks_project_id_projects` |
| Enum type | `snake_case` | `document_status` |

## Migration Order

Migrations must be applied in dependency order. Tables with no FKs first, then dependent tables.

### Batch 1 — No Dependencies
1. `projects`
2. `agents`
3. `llm_settings`

### Batch 2 — Depend on Batch 1
4. `requirement_inputs` (→ projects)
5. `tasks` (→ projects, agents)
6. `pipeline_runs` (→ projects)
7. `documents` (→ projects, agents)
8. `sprints` (→ projects)
9. `milestones` (→ projects)

### Batch 3 — Depend on Batch 2
10. `pipeline_steps` (→ pipeline_runs, agents, documents)
11. `messages` (→ projects, tasks)
12. `activity_logs` (→ projects, tasks, agents)
13. `traceability_links` (→ projects)
14. `agent_memories` (→ agents, projects)

## Running Migrations

```bash
# Apply all pending migrations
alembic upgrade head

# Roll back one migration
alembic downgrade -1

# Roll back to specific revision
alembic downgrade 0001

# Check current revision
alembic current

# Show migration history
alembic history --verbose

# Generate a new migration (auto-detect changes)
alembic revision --autogenerate -m "add_column_to_documents"
```

## Adding Enum Values — Special Procedure

PostgreSQL does not allow removing values from an enum type, and adding values requires care.

**To add a value to an existing enum:**
```python
# In Alembic migration file
def upgrade():
    op.execute("ALTER TYPE document_type ADD VALUE 'new_type' AFTER 'uat_script'")

def downgrade():
    # Downgrade cannot remove enum values — document this limitation
    pass  # manual intervention required
```

> **Warning:** Adding enum values is irreversible in PostgreSQL. Verify the value name carefully before applying.

## Rollback Strategy

| Scenario | Action |
|---|---|
| Failed migration on local | `alembic downgrade -1`, fix the migration, re-apply |
| Failed migration on staging | Restore from pre-migration DB snapshot, fix migration, re-deploy |
| Failed migration on production | Restore from backup immediately, investigate root cause, apply during next maintenance window |
| Data corruption | Restore from point-in-time backup (requires PostgreSQL WAL archiving) |

**Before any production migration:**
1. Create a full database dump: `pg_dump ai_sdlc > backup_$(date +%Y%m%d_%H%M%S).sql`
2. Apply migration in a transaction where possible
3. Verify row counts in affected tables match expectations
4. Keep the backup for 30 days minimum
